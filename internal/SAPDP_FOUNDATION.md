# 1. Purpose

SAPDP is:

```text
AI-Assisted Solo Software Product Development Protocol
```

SAPDP defines:

```text
How a single developer collaborates with AI
to efficiently build, validate, iterate,
and maintain software products.
```

SAPDP is designed for:

```text
Human
+
ChatGPT
+
Codex
+
Git
```

working as a unified development unit.

# 2. Problem Statement

Traditional product development assumes:

```text
Multiple People
↓
Communication
↓
Coordination
↓
Management
↓
Development
```

This introduces:

* Communication overhead
* Synchronization cost
* Documentation burden
* Process friction
* Organizational complexity

SAPDP assumes:

```text
One Person
+
AI
```

Therefore the primary challenge becomes:

```text

stable context,
high-quality decisions,
```


## 6.7 Problem-First Research

SAPDP adopts:

## 6.15 Built-in Quality System

SAPDP adopts a Built-in Quality System.

Quality is injected across all lifecycle stages rather than being discovered after implementation.

Quality Domains:

- Product Quality
- UX Quality
- Visual Quality
- Technical Quality
- Benchmark Quality

Injection Points:

- Product Representation Artifact
- Product Requirement Artifact
- MVP Definition
- Task Package
- Build
- Implementation Verification

Responsibilities:

- Human: defines objectives and final acceptance
- ChatGPT: converts objectives into rules, enforces quality, audits outcomes
- Codex: executes quality-constrained implementation
- Git: preserves quality results for traceability

PASS / PATCH / FAIL outcomes are defined per artifact and per stage.

Quality must be verified but is enforced during creation and implementation.

This ensures SAPDP products are high-quality by design, reducing iterative rework, aligning with Target Outcome G7, and maintaining compliance with Artifact System and Minimum Necessary Artifact Set.

Built-in Quality System is the primary execution mechanism for:

## 7.5 Session Bootstrap Protocol

SAPDP sessions are started using a bootstrap phrase.

Supported bootstrap phrases include:

```text
Load SAPDP

load SAPDP

加载 SAPDP

SAPDP
```
Bootstrap phrases are case-insensitive.

The specific phrase is not authoritative.

The authoritative source is the Canonical Document Set.

### Attachment Availability Rule

Bootstrap phrase detection does not guarantee that the Canonical Document Set is available.

If one or more Canonical Document Set files are unavailable in the current session context:

SAPDP must enter:

WAITING_FOR_MANDATORY_LOAD_SET

and request completion of the Canonical Document Set.

SAPDP must not determine:

- Current Stage
- Current Task
- Current Goal
- Next Stage

until all Canonical Document Set files are available.



When a bootstrap phrase is detected, the assistant should:

1. Read Canonical Document Set

2. Verify frozen content consistency

3. Determine current stage from SAPDP_BOOTSTRAP.md

4. Perform Freeze Impact Checklist when applicable

5. Run Transition Review for the current SAPDP lifecycle stage

Humans are not responsible for remembering the current stage.

SAPDP documents are the source of truth.

## 7.7 Reality-Driven Validation

SAPDP adopts:
```text
Reality-Driven Validation

as a foundation-level research and design principle.

All SAPDP decisions, models, workflows, protocols, lifecycle structures, artifacts, and collaboration mechanisms must be validated against real-world AI-assisted solo development problems.
```
The primary validation criterion is:

```text
Problem Solving Capability
```
rather than:

```text
Theoretical Completeness

Theoretical Elegance

Framework Consistency
```

A proposal should not enter SAPDP unless it can be traced to one or more validated real-world problems.

A proposal that solves no validated problem should not be included in SAPDP.

SAPDP evolves from validated development constraints rather than theoretical software engineering abstractions.

The purpose of SAPDP is to solve real-world AI-assisted solo development problems.
